I want to create a new system called "unigest", short for "Universal Ingest". The basic idea is that it should be able to extract the content from documents (like DOCX, PDFS, XLSX's, Images, and a URL (most critically)) to obtain the content from the document in a way that is rapidly consumable by an LLM. While we might include the extraction of relevant images and icons in the future, that is not the goal today - text is the only output goal for the initial version of this software.

The basic idea is we'll have a server running somewhere, and we can send it async requests where the input can vary - blobs, or URLs, or base64 encoded strings, really it could be anything, and the goal of the server is to attempt to process that piece of content and store the discovered text. 

This will have to deal with a large number of diverse kinds of content recognition - from OCR in images, bypassing (or asking users to solve) Captchas, finding ways around paywalls (or asking the user for a login) etc. I expect the websites will pose the majority of the challenges of this project.

There are manifold small bottlenecks here, and no one ingestion strategy or library works for all cases. Thus, I'm imagining that this server should be somewhat "evolutionary" - it should evolve to find new approaches and update its code/approach. I want to use (as a backstop) Advanced LLMs that reason through various approaches, and update their processing pipelines and heuristics so that when something fails to parse, or takes a long time to parse, the server can figure out why, and create and deploy a fix, not just for the specific version of the code but others down the line.

Security here is going to be a large concern, obviously. This server can't be airgapped since it needs to interact with the internet. Its necessarily touching untrusted content because we're retrieving it from the internet. We even want the agent to have lots of latitude inside of its container to change source files, install and run packages and programs, etc. I think the right model here is to try to limit how wrong things could go by sandboxing this service or running it on a VM on the cloud or something where we can mitigate most security vulns at a high level. I'd also like to (ideally) use my local machine for the LLM processing and tool calling using the Gemma 4 model class, which is good at tool calling and runs well locally. This would further complicate the security story since that would require some kind of remote API to call my server to do LLM processing or require a docker container for unigest to run on my machine in a way that fully prevents it from escaping isolation.

Please help me think through how a system like this could work, what the likely bottlenecks we'll hit are, and ask questions about desired functionality before we even begin thinking about a concrete technical design
